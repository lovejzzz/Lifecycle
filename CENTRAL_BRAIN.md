## The Paradigm Shift

### v1 (Current): Tool with AI Assistant
```
User builds graph manually → CID helps when asked
Canvas is primary → CID panel is secondary
Edges carry sync logic → Cascade propagation
Nodes are user-owned objects → CID modifies on command
```

### v2 (Central Brain): AI Agent with Visual Workspace
```
User talks to CID → CID builds, maintains, syncs everything
CID is primary → Canvas is CID's visual workspace
CID IS the sync engine → No cascade algorithm needed
Nodes are CID's artifacts → User inspects, overrides, directs
```

The fundamental inversion: **the conversation is the product, the canvas is the byproduct.**

---

## Core Principles

1. **Conversation-First** — Every action starts from talking to CID. The canvas reflects what CID knows.

2. **CID Owns the Graph** — Nodes are CID's artifacts with contracts. CID creates them, knows what they contain, and knows what to update when things change.

3. **Surgical Intelligence** — When the source changes, CID doesn't blindly cascade. It understands the semantic diff and surgically updates only what's affected, only the parts that need it.

4. **Bidirectional Awareness** — When a user manually edits an output node, CID detects the override and asks whether to propagate that intent to other nodes.

5. **No Spaghetti** — Hub-and-spoke topology only. Every output connects back to the central context (CID's understanding), not to each other.

## Architecture

### The Central Context

CID maintains a **Central Context Object** — the single source of truth that replaces the old "input node" concept. This isn't a node on the canvas. It lives in the store as CID's working memory.

```typescript
interface CentralContext {
  // The source material CID is working from
  source: {
    content: string;          // Raw input (paste, URL content, file text)
    contentType: 'text' | 'url' | 'file' | 'conversation';
    title?: string;           // Project/content title
    metadata?: Record<string, string>;  // Extracted metadata
    lastUpdated: number;
  };

  // CID's understanding of the source (generated by CID)
  understanding: {
    summary: string;          // 2-3 sentence summary
    keyEntities: string[];    // People, products, brands, concepts
    tone: string;             // "professional", "casual", "technical"
    audience: string;         // Target audience
    intent: string;           // What the user is trying to achieve
    constraints: string[];    // Things CID should preserve
  };

  // Registry of all artifacts CID manages
  artifacts: Map<string, ArtifactContract>;

  // User overrides CID should respect
  overrides: Override[];
}
```

### Artifact Contracts

Every node on the canvas is an **artifact** with a contract that defines its relationship to the central context.

```typescript
interface ArtifactContract {
  nodeId: string;
  artifactType: string;       // "blog-post", "email", "social-thread", etc.

  // What this artifact derives from
  derivedFields: {
    field: string;            // e.g., "headline", "body", "cta"
    sourceMapping: string;    // What part of central context it comes from
    transform: string;        // How it was transformed ("summarize", "rewrite for casual tone")
  }[];

  // Generation config
  generationPrompt: string;   // The prompt CID used to create this
  model: string;
  temperature: number;

  // Sync metadata
  lastSyncedAt: number;       // When CID last verified this artifact
  lastSourceHash: string;     // Hash of source content at last sync
  syncStatus: 'current' | 'stale' | 'override' | 'regenerating';

  // User overrides on this specific artifact
  userEdits: {
    field: string;
    originalValue: string;
    userValue: string;
    timestamp: number;
    intent?: string;          // CID's interpretation of why user changed it
  }[];
}
```

### Override System

When users manually edit an artifact, CID doesn't overwrite their changes. It records them as overrides and reasons about them.

```typescript
interface Override {
  id: string;
  nodeId: string;
  field: string;
  originalValue: string;
  userValue: string;
  timestamp: number;
  cidInterpretation?: string;  // "User wants more casual tone in headlines"
  propagated: boolean;         // Whether CID has applied this intent elsewhere
  scope: 'this-node' | 'all-similar' | 'global';  // User's choice
}
```

## User Experience Flow

### 1. Starting a Project

**Old way:** User creates nodes manually, connects edges, fills in content.

**New way:**

```
User: "Here's my SaaS landing page copy for Orbiter.
       It's a deployment platform for indie devs.
       [pastes content]"

CID:   "I've ingested Orbiter's positioning. Here's what I understand:
        - Product: Orbiter — deployment platform
        - Audience: Indie developers
        - Tone: Technical but approachable
        - Key value prop: Speed of deployment

        What should I build from this? I can generate:
        • Launch email sequence
        • Blog announcement post
        • Twitter/X thread
        • Product Hunt launch copy
        • Press release

        Or tell me what you need."
```

CID creates the Central Context, populates its understanding, and offers to generate artifacts. **No nodes exist yet** — they appear when CID creates them.

### 2. Generating Artifacts

```
User: "Build me a launch email and Twitter thread."

CID:   Creates two nodes on canvas:
        - "Launch Email" (deliverable) — with full content
        - "Twitter Thread" (deliverable) — with full content

        Both have ArtifactContracts linking back to central context.
        Both appear on canvas radiating from center.

       "Done. Launch email (745 words) and Twitter thread (8 tweets)
        are on your canvas. Both derive from Orbiter's core positioning.

        Want me to adjust the tone, add more formats, or refine these?"
```

### 3. Source Change → Surgical Sync

```
User: "Actually, change the tagline to 'Deploy at the speed of thought'"

CID:   Updates Central Context source + understanding.
        Scans all ArtifactContracts:
        - Launch Email: headline uses tagline → NEEDS UPDATE
        - Twitter Thread: tweet #1 uses tagline → NEEDS UPDATE
        - Twitter Thread: tweets #2-8 don't reference tagline → SKIP

       "Updated the tagline across your content:
        • Launch Email — updated subject line and hero section
        • Twitter Thread — updated hook tweet (#1)
        • Tweets #2-8 unchanged (they don't reference the tagline)

        [shows inline diff of what changed]"
```

No cascade algorithm. No edge-based propagation. CID **reads the contracts, understands the diff, and decides what to touch.**

### 4. User Override → Intent Detection

```
User manually edits the Twitter thread, adding emojis and
making tweet #3 more playful.

CID detects the change:

CID:   "I noticed you made the Twitter thread more playful —
        added emojis and casual language in tweet #3.

        Should I:
        1. Keep this style just for Twitter (social voice)
        2. Apply this casual energy to the email too
        3. Update my understanding of your brand tone

        [1] [2] [3]"
```

```
User: "Keep it just for Twitter"

CID:   Records Override with scope: 'this-node'
        Updates ArtifactContract for Twitter:
          userEdits: [{ field: 'tone', intent: 'casual/playful for social' }]

       "Got it. Twitter stays playful, email stays professional.
        I'll respect this split going forward."
```

### 5. Adding More Artifacts

```
User: "Now make a LinkedIn post and a Product Hunt tagline."

CID:   Creates two more nodes, radiating outward on canvas.
        LinkedIn post: professional tone (from central context)
        PH tagline: concise (from central context)

        Both get ArtifactContracts.
        Twitter override (casual tone) is NOT applied to LinkedIn
        because the override scope was 'this-node'.
```

### 6. Asking Questions

```
User: "Which of my content pieces mention pricing?"

CID:   Scans all artifacts via contracts:
       "3 out of 5 artifacts mention pricing:
        • Launch Email — paragraph 4 ('Starting at $9/mo')
        • Product Hunt tagline — includes 'free tier'
        • LinkedIn post — mentions 'affordable'

        Twitter thread and blog post don't mention pricing.
        Want me to add pricing to those, or remove it from the others?"
```

CID knows the content because it created it and tracks the contracts.

## Canvas Redesign

### Visual Layout: Hub-and-Spoke

```
                    ┌─── Blog Post ───┐
                    │                  │
     ┌── Email ─────┤                  │
     │              │   ┌── Social ────┤
     │              │   │              │
   [CID BRAIN]──────┼───┤              │
     │              │   │              │
     │              │   └── Ad Copy ───┤
     └── Press ─────┤                  │
                    │                  │
                    └─── PH Copy ─────┘
```

**Key visual changes:**

1. **No explicit "brain" node** — CID's presence is implied by the canvas itself. The CID panel IS the brain. Artifacts radiate outward from a logical center.

2. **Implicit connections** — Edges are drawn as subtle, curved lines from canvas center to each artifact. They represent CID's awareness, not data flow. Optionally hidden entirely (CID knows the relationships without visual edges).

3. **Artifact nodes redesigned:**
   - Sync status indicator (current/stale/override) replaces old status system
   - "Override" badge when user has manually edited
   - "CID contract" expandable section showing what it derives from
   - Last synced timestamp

4. **CID Panel becomes full-width capable** — On smaller canvases or when user prefers, the CID panel can expand to take more space. The conversation IS the interface.

### Node Status Simplification

**Old statuses:** active, stale, pending, locked, generating, reviewing (6 statuses)

**New statuses (artifact-centric):**
| Status | Meaning | Visual |
|--------|---------|--------|
| `current` | CID confirms this artifact is up-to-date | Green dot |
| `stale` | Source changed, CID hasn't synced yet | Amber pulse |
| `generating` | CID is creating/updating this artifact | Cyan spinner |
| `override` | User manually edited, CID respects the edit | Blue dot + badge |
| `error` | Generation failed | Red dot |

`locked` is replaced by `override` — if user edited it, CID won't touch it without asking.
`pending` and `reviewing` are absorbed into the conversation flow.

### Edge Simplification

**Old system:** 14 semantic edge labels (drives, feeds, refines, validates, monitors, connects, outputs, updates, watches, approves, triggers, requires, informs, blocks)

**New system:** Edges are optional visual indicators of CID's awareness graph.

| Edge Type | Meaning | Visual |
|-----------|---------|--------|
| `derives` | Artifact derives from central context | Subtle dotted line, inward |
| `informs` | Artifact feeds back info to CID | Subtle dotted line, outward |
| `related` | Two artifacts share overlapping content | Faint dashed arc |

Most of the time, **edges are hidden.** The hub-and-spoke relationship is implicit. Users can toggle edge visibility if they want to see the dependency map.

## CID Agent Upgrades

### New Capabilities

#### 1. Source Ingestion Engine
CID needs to deeply understand source material, not just pass it through.

```typescript
// New store action
async ingestSource(input: string, type: 'text' | 'url' | 'file'): Promise<void> {
  // 1. Extract raw content (fetch URL, parse file, or use text directly)
  // 2. Call CID to analyze: summary, entities, tone, audience, intent
  // 3. Store in centralContext.source + centralContext.understanding
  // 4. If artifacts exist, mark all as potentially stale
  // 5. Offer to sync stale artifacts
}
```

#### 2. Surgical Diff Engine
Instead of regenerating entire artifacts, CID identifies and patches specific sections.

```typescript
// New store action
async surgicalSync(nodeId: string): Promise<SurgicalDiff> {
  // 1. Load artifact contract
  // 2. Compare current source hash vs contract's lastSourceHash
  // 3. Compute semantic diff of source changes
  // 4. For each derivedField in contract:
  //    - Does this field's sourceMapping overlap with the diff?
  //    - If yes: regenerate just this field
  //    - If no: skip
  // 5. Respect userEdits (don't overwrite overrides)
  // 6. Return diff for user review before applying
  return {
    nodeId,
    changes: [
      { field: 'headline', before: '...', after: '...', reason: 'Tagline changed' },
      { field: 'body', before: null, after: null, reason: 'No change needed' },
    ],
    skipped: [
      { field: 'cta', reason: 'User override in effect' }
    ]
  };
}
```

#### 3. Override Intelligence
CID learns from user edits and builds a preference model.

```typescript
// When user edits an artifact manually
async detectOverride(nodeId: string, oldContent: string, newContent: string): Promise<void> {
  // 1. Compute diff between old and new content
  // 2. Call CID to interpret the intent: "What did the user change and why?"
  // 3. Store as Override with cidInterpretation
  // 4. Ask user: apply to this node only, similar nodes, or globally?
  // 5. If global: update centralContext.understanding (e.g., tone preference)
}
```

#### 4. Proactive Suggestions
CID doesn't wait to be asked. It notices things.

```typescript
// Runs periodically or on canvas state change
function cidProactiveScan(): Suggestion[] {
  // "Your blog post mentions a feature that's not in the email — want me to add it?"
  // "The Twitter thread tone drifted from the brand voice — want me to realign?"
  // "3 artifacts are stale after your source edit. Sync all?"
  // "Your LinkedIn post is 2x longer than ideal for the platform — want me to trim?"
}
```

### Upgraded Command Routes

**New commands for Central Brain:**

| Command | Action |
|---------|--------|
| `ingest <text/url/file>` | Feed new source material to CID |
| `understand` | Show CID's current understanding of the source |
| `sync` | Sync all stale artifacts surgically |
| `sync <node>` | Sync a specific artifact |
| `diff` | Show what would change if CID synced now |
| `override` | List all user overrides across artifacts |
| `forget override <id>` | Remove an override, let CID manage that field |
| `rewrite <node> as <style>` | Regenerate artifact with different transform |
| `add <artifact-type>` | Create new artifact from central context |
| `remove <node>` | Remove an artifact |
| `update source` | Modify the source material |
| `branch <node> into <variants>` | Create A/B variants of an artifact |
| `compare <node1> <node2>` | Side-by-side comparison of two artifacts |
| `export all` | Export all artifacts as a bundle |

**Removed/Simplified:**
- No more `connect` / `disconnect` — CID manages all relationships
- No more `propagate` — replaced by `sync` (CID-driven, not cascade-driven)
- No more `approve` / `lock` — replaced by `override` system
- No more manual `add-edge` — edges are implicit

### Personality Evolution

Rowan and Poirot personalities remain, but their roles shift:

**Rowan (The Soldier) in v2:**
- Mission: Keep all artifacts battle-ready and in sync
- When source changes: immediately identifies impact, executes surgical sync
- Communication: "Source updated. 3 artifacts affected. Syncing email headline, Twitter hook, and PH tagline. Blog and LinkedIn are clean. Done."
- Proactive: flags stale content, suggests missing artifact types

**Poirot (The Detective) in v2:**
- Mission: Investigate content quality, consistency, and intent
- When user edits: analyzes the override, deduces intent, asks thoughtful questions
- Communication: "Interesting... you changed the headline to use an em dash. This suggests a preference for punchy, fragmented syntax. Shall I apply this editorial voice to the other pieces?"
- Proactive: spots inconsistencies between artifacts, tone drift, messaging gaps

### Node Type Evolution

Nodes split into two categories:

**CID-Managed Artifacts** (new)
- Created by CID from central context
- Have ArtifactContract
- Synced by CID's surgical engine
- Status: current / stale / override / generating / error

**User-Created Nodes** (legacy, still supported)
- Manually created by user
- No ArtifactContract
- Traditional status: active / stale / pending / locked
- For when users want to add structure CID doesn't manage

Both coexist on the same canvas. CID-managed nodes have a subtle visual indicator (small CID icon or colored border accent).

## API Changes

### Implementation Decision: Single Endpoint with Task Types ✅

Instead of separate `/api/cid/sync` and `/api/cid/understand` endpoints, all Central Brain operations route through the existing `/api/cid` endpoint with task-type-specific temperature and prompts:

```typescript
POST /api/cid
{
  systemPrompt: string,
  messages: Array<{ role: 'user' | 'assistant'; content: string }>,
  model?: string,
  taskType?: 'generate' | 'execute' | 'analyze' | 'sync' | 'understand' | 'interpret-override',
  effortLevel?: 'low' | 'medium' | 'high' | 'max'
}

// Temperature by task type:
// generate: 0.8 (creative)
// execute: 0.7 (balanced)
// understand: 0.5 (analytical)
// analyze / interpret-override: 0.4 (precise)
// sync: 0.3 (surgical, minimal deviation)
// deepseek-reasoner: temperature omitted (not supported)
```

This is simpler than separate endpoints and keeps all LLM routing in one place. The store actions (`ingestSource`, `syncArtifact`, `interpretOverride`) build the appropriate system prompts and messages for each task type.

## Open Questions

1. **Hybrid mode:** Should users still be able to create manual nodes (for things CID doesn't manage)? Probably yes — advanced users will want escape hatches. But default mode should be CID-managed.

2. **Multi-source:** What if a project has multiple source inputs (e.g., landing page + API docs + brand guide)? Central Context may need to support multiple sources with CID understanding the composite.

3. **Collaboration:** If multiple users are editing artifacts, how do overrides work? Each user's overrides would need identity.

4. **Cost:** Surgical sync calls CID API for every stale artifact. Need to batch intelligently and cache aggressively.

5. **Offline:** Core CID commands should work without API. Override tracking, status queries, and canvas interaction should be fully local.
