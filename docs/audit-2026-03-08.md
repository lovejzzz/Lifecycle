## 1. Architecture — Grade: B+

### Findings

1. **Well-extracted library modules.** Core logic is split into focused files with pure functions:
   - `/src/lib/graph.ts` — layout, topological sort (Kahn's algorithm), edge utilities, markdown-to-HTML converter (177 lines)
   - `/src/lib/intent.ts` — NLP intent detection, service/file type knowledge bases, node generation from prompts
   - `/src/lib/agents.ts` — agent personalities, response templates, interview system configuration
   - `/src/lib/prompts.ts` — system prompt builder with graph serialization, conversation compression (203 lines)
   - `/src/lib/types.ts` — shared types, color system with custom category registration, icon registry (303 lines)

2. **Server-side AI routing is sound.** `/src/app/api/cid/route.ts` (309 lines) handles three providers (DeepSeek, Anthropic, OpenRouter) with automatic fallback, model-specific parameters (temperature omitted for reasoner, 16,384 max_tokens for reasoner vs 4,096 for others), and robust JSON extraction including brace-counting for malformed LLM output (lines 161-172).

3. **Comprehensive normalization layer.** The API route normalizes LLM output variations:
   - Edge labels: 14 known + 19 synonyms mapped (lines 179-189)
   - Node categories: 14 known + 60+ synonyms mapped (lines 192-225)
   - Alternate key names: `source`/`target` to `from`/`to`, `name`/`title`/`node_name` to `label` (lines 238, 281)
   - Alternate edge containers: `connections`/`links`/`relationships` to `edges` (line 229)
   - String-to-index ID resolution with fallback to node ID lookup (lines 252-262)
   - Smart edge label inference from source/target category pairs (lines 264-275)

4. **Conversation memory compression.** `buildMessages()` in `prompts.ts` (lines 149-203) compresses conversations over 10 messages into semantic summaries, extracting user intents and CID actions. Keeps token budget bounded while preserving context.

5. **Monolithic store is the main architectural weakness.** `/src/store/useStore.ts` at 4,481 lines contains all state, UI logic, workflow operations, AI orchestration, localStorage persistence, animation timers, and template management in one file.

6. **Dual agent system is cleanly separated.** Agent personalities (Rowan/Poirot) defined in `agents.ts` with distinct response templates. Rowan skips interviews and is action-first; Poirot uses a multi-question investigation system. Both share the same engine and see the same graph state.

### Recommendations
- Split `useStore.ts` into domain slices: `graphStore`, `cidStore`, `uiStore`, `persistenceStore`.
- Extract `executeNode` and `executeWorkflow` into a dedicated `execution.ts` module.
- Consider a middleware pattern for cross-cutting concerns (logging, persistence).

## 2. Security — Grade: B

### Findings

1. **API keys are server-side only.** All LLM traffic routes through `/api/cid`. The `apiKey` field on `NodeData` is marked `@deprecated` (`types.ts` line 58) and actively stripped on save (`useStore.ts` line 935: `apiKey: undefined`). Correct security posture.

2. **Input validation on the API route.** Validates `systemPrompt` and `messages` are present and `messages` is a non-empty array (`route.ts` lines 16-21). Returns 400 for invalid requests, 503 when no API keys configured.

3. **No rate limiting on `/api/cid`.** The route has retry logic for upstream 429 responses (lines 127-139) but no inbound rate limiting. A malicious or buggy client can trigger unlimited upstream API calls and spend.

4. **No CORS configuration.** The API route does not set explicit CORS headers. While Next.js defaults are reasonable for same-origin, no explicit policy is documented or enforced.

5. **No CSRF protection.** The POST endpoint does not validate origin or include CSRF tokens. Next.js Server Actions have built-in CSRF protection, but raw route handlers do not.

6. **No input sanitization before LLM injection.** User messages pass directly into the LLM prompt without sanitization for prompt injection. The 72-line system prompt constrains output format but does not defend against adversarial input.

7. **No `dangerouslySetInnerHTML` found.** The codebase avoids direct HTML injection. `markdownToHTML` in `graph.ts` (lines 155-176) escapes `&`, `<`, `>` before applying markdown transformations.

8. **localStorage stores workflow data in plain text.** Workflow content, messages, and custom templates persisted without encryption. Sensitive information in node content (API endpoints, credentials, etc.) would be readable by any script on the same origin.

### Recommendations
- Add rate limiting middleware to `/api/cid` (e.g., 30 requests/minute per IP).
- Add input length limits: cap `systemPrompt` at 50KB, each message at 10KB.
- Add CSRF validation or migrate to Server Actions.
- Add Content-Security-Policy headers.

## 3. Performance — Grade: B-

### Findings

1. **Component memoization is present but limited.** 25 total `useMemo`/`useCallback`/`memo` occurrences across 5 component files. `Canvas.tsx` has 19 occurrences across 1,362 lines — good optimization of the main render surface.

2. **Selective Zustand selectors in LifecycleNode.** The node component uses individual selectors (lines 29-34) and a custom edge-counting selector (lines 36-42) that iterates edges once and avoids re-renders on unrelated changes. Well-designed.

3. **Large store risks unnecessary re-renders.** With 4,481 lines in a single store, `set()` calls for high-frequency updates (streaming messages, animation timers) share the same store instance as low-frequency state (templates, rules). Any connected component must evaluate its selectors on every update.

4. **Staggered animation creates many sequential state updates.** Node/edge build animation uses `setTimeout` chains (`useStore.ts` lines 2147-2198): `200 + i * 300` ms per node, `120` ms per edge. A 10-node/12-edge workflow generates ~22 individual `set()` calls over ~3.5 seconds.

5. **DeepSeek Reasoner timeout is 240 seconds.** (`route.ts` line 124). Eval report confirms generation tasks take 28-93 seconds. No user-facing progress indicator during these waits beyond a spinner.

6. **No lazy loading of heavy panels.** `CIDPanel` (1,423 lines), `ActivityPanel`, `NodeDetailPanel`, and `NodeContextMenu` are all eagerly imported in `Canvas.tsx`.

7. **localStorage persistence correctly filters ephemeral messages.** (`useStore.ts` line 450: `messages.filter(m => !m._ephemeral)`). Streaming/thinking messages do not bloat storage.

### Recommendations
- Split store to isolate high-frequency updates from graph state.
- Use `React.lazy` for `CIDPanel`, `ActivityPanel`, `NodeDetailPanel`.
- Batch node/edge additions during build animation to reduce re-render count.
- Add debounced localStorage persistence instead of saving on every state change.

## 4. Code Quality — Grade: B+

### Findings

1. **Strong TypeScript typing.** `NodeData` interface (`types.ts` lines 40-71) is comprehensive with JSDoc. `CIDMessage`, `LifecycleEvent`, `AgentPersonality`, and `AgentResponses` interfaces are well-defined. `NodeCategory` is typed as `string` (line 4) to allow custom categories — a pragmatic extensibility choice.

2. **DRY edge/node helpers.** `createStyledEdge()` (`graph.ts` lines 81-94), `inferEdgeLabel()` (lines 97-126 with 40+ category pair mappings), and `findNodeByName()` (lines 131-136 with fuzzy matching) are reusable pure functions.

3. **Comprehensive normalization maps.** The API route defines 4 normalization systems: `KNOWN_LABELS` (14 entries), `LABEL_MAP` (19 synonyms), `KNOWN_CATEGORIES` (14 entries), `CATEGORY_MAP` (60+ synonyms). These handle real-world LLM output variations effectively.

4. **Long functions in the store.** `chatWithCID` spans ~150 lines, `executeNode` spans ~145 lines, both mixing UI state updates, API calls, animation scheduling, and error handling.

5. **Edge label constants defined in three places.** `Canvas.tsx` line 35 (`EDGE_LABELS` array), `route.ts` line 179 (`KNOWN_LABELS` Set), and `graph.ts` line 78 (`ANIMATED_LABELS` Set). Should consolidate to a single source of truth in `types.ts`.

6. **`markdownToHTML` is fragile.** The regex-based markdown converter (`graph.ts` lines 155-176) handles basic headings, bold, italic, code, lists, blockquotes, and horizontal rules. Would break on nested lists, fenced code blocks, or multi-line constructs. Used for PDF/HTML export.

7. **Consistent code style.** Files use consistent formatting, import organization, and comment section headers (`// ---` separators). Prompt constants use clear naming (`SHARED_CAPABILITIES`, `ROWAN_PERSONALITY`, `POIROT_PERSONALITY`).

### Recommendations
- Extract `chatWithCID` and `executeNode` into separate modules.
- Consolidate edge label constants into `types.ts` as the single source of truth.
- Replace regex markdown converter with a lightweight library (`marked` or `markdown-it`).

## 5. Error Handling — Grade: B

### Findings

1. **Structured API error responses.** Returns `{ error: 'invalid_request', message }` with 400, `{ error: 'no_api_key' }` with 503, `{ error: 'api_error', message }` with 500 (`route.ts` lines 17-21, 29-32, 144-147, 304-307).

2. **Retry with exponential backoff for 429s.** Up to 4 attempts with `2^(attempt+1) * 1500ms` backoff (`route.ts` lines 127-139). Wait times: 3s, 6s, 12s, 24s.

3. **Graceful JSON parse fallback.** When LLM response is not valid JSON, returns it as a plain message with `workflow: null` (`route.ts` lines 298-299). Prevents crashes on malformed output.

4. **`executeNode` handles errors at three levels.** HTTP status check (line 919), API error in response body (line 926), and caught exceptions (line 939). Each path updates `executionStatus` and `executionError` on the node.

5. **`chatWithCID` catch block swallows error context.** (Line 2233) Uses `_err` (discarded) and falls back to template response. The actual error is not logged, making production debugging difficult.

6. **localStorage operations silently fail.** Save errors caught with `console.warn` (line 455). Rule persistence: `catch { /* ignore */ }` (line 545). Template persistence: empty catch (line 3938).

7. **No AbortController for client-side fetches.** `executeNode` and `chatWithCID` do not use AbortController. Users cannot cancel in-flight requests, and navigating away leaves orphaned requests.

### Recommendations
- Log errors in catch blocks before falling back to templates.
- Add AbortController to all client-side fetch calls.
- Surface localStorage quota errors via toast notifications.
- Add timeout feedback: show "Request timed out" after 60s+ waits.

## 6. State Management — Grade: C+

### Findings

1. **Zustand is well-suited.** Uses `get()` for async operations (avoiding stale closures) and individual selectors in components (minimizing re-renders). Version 5 with React 19 compatibility.

2. **Single store at 4,481 lines.** Contains:
   - Graph state (nodes, edges, events)
   - UI state (selected node, panels, context menus, search, multi-select)
   - Agent state (CID mode, Poirot interview context)
   - Undo/redo history (full graph snapshots)
   - AI orchestration (executeNode, executeWorkflow, chatWithCID, generateNodeContent)
   - Persistence (localStorage save/load with ephemeral filtering)
   - Animation state (staggered timers, streaming)
   - Templates, rules, model picker, toast notifications

3. **Undo/redo captures full graph snapshots.** `history` and `future` arrays store complete `{ nodes, edges }` copies. No bounded limit visible in the code read. For large workflows, this accumulates significant memory.

4. **Animation timer management is clean.** `trackTimeout`/`clearAnimationTimers` pattern (line 2142) prevents timer leaks when a new build starts before the previous animation finishes.

5. **No derived state caching.** Stale node count, orphan detection, and connection counts are recomputed on each access. These could be memoized selectors.

6. **Multi-select uses `Set<string>`.** Correct for Zustand's immutable-friendly API.

### Recommendations
- Split into 3-4 Zustand slices with clear domain boundaries.
- Implement memoized derived selectors: `useStaleNodes()`, `useOrphanNodes()`.
- Bound undo history (max 50 snapshots).
- Extract animation/streaming into ephemeral store that does not trigger persistence.

## 7. Accessibility — Grade: D

### Findings

1. **Zero ARIA attributes across all components.** No `aria-label`, `aria-describedby`, `aria-live`, `role`, or `aria-expanded` found in any component. 0 matches across all files in `/src/components/`.

2. **No keyboard navigation for custom UI.** CID Panel commands, node context menu, edge label picker, card selection, and autocomplete dropdown appear mouse-only. No `tabIndex`, `onKeyDown`, or focus management detected.

3. **Color-only status indicators.** Node status (`LifecycleNode.tsx` lines 13-20) uses color as the primary differentiator: green (active), yellow (stale), purple (pending), gray (locked), cyan (generating), red (reviewing). Some have icons (AlertTriangle, Lock, Loader2) but not all. Fails WCAG 1.4.1 (Use of Color).

4. **No focus trap in modals/panels.** CID Panel and NodeDetailPanel open as sidebars without focus trapping. Keyboard users can tab behind them.

5. **No screen reader announcements for async operations.** AI generation (28-93 seconds), workflow building, and node execution produce no `aria-live` announcements.

6. **React Flow provides baseline canvas accessibility.** Keyboard pan/zoom and node selection are available through the library, but all custom UI layered on top lacks accessibility support.

### Recommendations
- Add `aria-label` to all interactive elements (icon buttons, inputs, toggles).
- Add `aria-live="polite"` regions for AI response streaming and build status.
- Implement keyboard navigation for CID Panel command list and card selection.
- Add focus trap to CID Panel and NodeDetailPanel.
- Add text labels alongside color for all status indicators.

## 8. Testing — Grade: C

### Findings

1. **38-test eval pool with structural and content scoring.** Located in `tests/eval/`. Tests validate LLM output quality with checks for:
   - Valid JSON response
   - Correct workflow/null based on task type (generate vs. analyze)
   - Node count bounds (min 5, max 10)
   - Required category presence (review, test, etc.)
   - Edge validity (indices in bounds)
   - Start/end node categories (input/trigger and output)
   - Description completeness (100% coverage)
   - Content depth (average character count, no thin nodes below threshold)
   - Graph connectivity (path from first to last node)

2. **Latest eval: 99% pass rate (6/6 selected tests).** Report at `tests/eval/2026-03-08T08-15-19/report.md`. All tests pass with DeepSeek Reasoner. Generation times: 28-93 seconds. Both Rowan and Poirot agents tested.

3. **No unit tests.** Zero `.test.ts` or `.spec.ts` files in the project source. No test runner (Vitest, Jest) in dependencies or devDependencies. No `"test"` script in `package.json`.

4. **Highly testable pure functions remain untested.**
   - `topoSort()` — Kahn's algorithm, 23 lines (`graph.ts` lines 51-74)
   - `inferEdgeLabel()` — 40+ category pair mappings (`graph.ts` lines 97-126)
   - `findNodeByName()` — fuzzy search with 3 strategies (`graph.ts` lines 131-136)
   - `analyzeIntent()` — NLP intent detection (`intent.ts` line 80+)
   - `buildMessages()` — conversation compression (`prompts.ts` lines 149-203)
   - `markdownToHTML()` — regex-based converter (`graph.ts` lines 155-176)
   - Normalization maps in `route.ts` (lines 179-295)

5. **No component tests.** No React Testing Library or end-to-end test framework. `Canvas.tsx` (1,362 lines), `CIDPanel.tsx` (1,423 lines), and `LifecycleNode.tsx` (410 lines) have no automated UI tests.

6. **CI check script covers types and lint only.** `"check": "npm run typecheck && npm run lint && npm run build"` catches type errors and lint violations but no runtime behavior.

### Recommendations
- Add Vitest and write unit tests for all pure functions in `graph.ts`, `intent.ts`, `prompts.ts`.
- Add integration tests for the normalization pipeline in `route.ts`.
- Add component tests with React Testing Library for LifecycleNode and CIDPanel.
- Add `"test"` script to `package.json` and include in `"check"` pipeline.
- Consider snapshot tests for normalization maps to catch regressions.

## What's Working Well

1. **Agent-first architecture.** CID drives the experience; the canvas renders the result. The 72-line shared capabilities prompt is detailed and effective. Rowan and Poirot have genuinely different behaviors (action-first vs. investigative interview), not just cosmetic differences.

2. **LLM output resilience.** The normalization layer in `route.ts` handles a remarkable range of output variations: 60+ category synonyms, 19 edge label synonyms, alternate key names, string-vs-numeric IDs, markdown-wrapped JSON, preamble text before JSON, and alternate container keys. The brace-counting JSON extractor (lines 165-172) is particularly robust.

3. **Conversation memory management.** Compression in `buildMessages()` keeps token usage bounded. Ephemeral messages (`_ephemeral: true`) prevent streaming artifacts from polluting localStorage. Welcome-back messages regenerated per session.

4. **Eval system design.** Structural + content scoring provides meaningful quality metrics: node count bounds, category presence, edge validity, content depth thresholds, graph connectivity verification. The 99% pass rate with DeepSeek Reasoner validates the prompt engineering.

5. **Clean library extraction.** The Round 68 extraction produced `graph.ts`, `intent.ts`, `agents.ts`, `prompts.ts`, and `types.ts` as well-separated, pure-where-possible, independently importable modules.

6. **Execution pipeline.** `executeNode` handles output nodes with multi-format file download (PDF via print dialog, HTML, JSON, plain text), passthrough for nodes without prompts, auto-prompt generation by category (12 category-specific prompt templates), upstream result collection, and proper status lifecycle (`generating` -> `active`/`error`).

7. **Provider abstraction.** Server resolves model name to provider (Anthropic, DeepSeek, OpenRouter) with automatic fallback chain. Task-aware temperature (0.8 for generation, 0.4 for analysis). Reasoner-specific handling (no temperature, 16K tokens, 240s timeout).

8. **Edge intelligence.** `inferEdgeLabel` maps 40+ category pair combinations to semantic labels. The API route adds a secondary inference layer for when LLM labels are generic or missing.
